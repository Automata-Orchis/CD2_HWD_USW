- **다중 페이지 PDF e2e** — `pypdfium2 scale=2.0` 분할의 한글 손글씨 가독성, N장 동시 입력의 VRAM/시간 증가 폭, 페이지 정보의 단일 JSON 통합 정확도 미실측.
- **모델 적재 wall-time 비교 (작업 선택 유무)** — "작업 선택 후 Analyze 자동 로드가 Load 버튼 단독보다 느리다"는 체감의 실재 여부 판정. 코드상 두 경로의 동시성 차이는 (a) 첫 신청서 PDF 의 이벤트 루프 동기 렌더링 1회성 + (b) 폴러 1→4개(초당 가벼운 SQLite 조회)뿐이며 18GB 가중치 디스크 읽기를 건드리지 않는다. `[qwen load] done in Ns` 로그 + 실측 VRAM 게이지로 두 경로의 실제 적재 시간을 측정 — 동일하면 (이전) 시간 추정 게이지의 착시, 유의미하게 다르면 I/O 경합 추적.

## 고려 사항

- **JSON 출력 강건화** — 이미지마다 모델 응답 형식이 달라 `_qwen_predict` 파서를 지속 보강. 현 상태: `max_new_tokens=1024` (512 잘림 이슈 해결) + 파싱 실패 시 `[qwen parse fail]` stdout 로그로 잘림/형식깨짐/빈응답 구분 가능. 부분 JSON 복구(잘림 시 마지막 닫히지 않은 `}` 자동 보정 등) 미도입.

## 개선 대상

- **DB 조회 GUI** — `python -m sqlite_utils` 가 불편하므로 별도 팝업 GUI 도입.
- **이미지 소스 전환** — 작업자 업로드 누적 방식 폐기, 서버 사전 저장 이미지를 frontend 에서 선택하는 구조로 전환.
- **신청서별 디렉토리 분류** — `server/data/<신청서종류>/` (예: `직불금신청서/`) 단위로 이미지 보관 (한글 경로 이슈 시 변경 여지).
- **작업 선택 UX** — frontend 에 신청서 종류별 진행도(전체/완료) 표시, 미완료 처리, 완료 신청서 재수정 흐름.
- **로그인 + 작업자 추적** — frontend 로그인으로 직원 소속/이름 수집 → DB `worker` 컬럼 기록.
- **DB 컬럼 확장** — 이미지 경로 / 추출 내용 / 작업 상태 / 작업자 / 작업 종료일 기본, 진행에 따라 추가.
- **이미지 N장 그룹핑 UX** — PDF 외 다중 이미지 신청서 그룹핑 미구현. PLAN.md §8 의 "Upload 1회 = 신청서 1건" 안 권장. 백엔드 DB/모델은 이미 신청서 단위라 `/upload` 와 frontend 업로드 핸들러만 손보면 된다.
- **표 형태 입력 정보 저장 방식** — 신청서에 표가 있을 경우 어떤 스키마/UI 로 저장할지 미정.
- **다행 표 필드 (세대원 등)** — `direct_payment` 의 세대원 / 분리세대 표처럼 한 항목에 여러 행이 들어가는 경우, 현재 단일 문자열로 평탄화되어 길게 나열됨. 엑셀 출력 스키마가 정해진 뒤 `field_spec` 에 list 타입 도입 또는 행별 키 (`relationship_rrc_1`, `_2`, …) 분리 검토.

## 추후 개선 대상

- **모델 캐시 정리** — 두 번째 VLM 어댑터 추가 직전, 이전 모델 `del` + `torch.cuda.empty_cache()` 의 "단일 VLM 슬롯" 정책 도입 (미도입 시 즉시 OOM).
- **Device 라우팅** — 어댑터가 `device_map="auto"` 고정이라 frontend 라디오가 no-op. `_run_job` 이 `req.device` 를 어댑터에 전달하도록 시그니처 확장.
- **Stop 입도** — 진행 중 단일 이미지 추론은 cancel 불가. 즉시 중단하려면 추론을 별도 프로세스로 분리 후 SIGTERM 종료.
- **frontend 폴링 종료** — terminal status(`completed`/`stopped`) 이후에도 `App.jsx` 의 `setInterval` 이 매초 3 GET 발생, 종료 분기 추가.
- **cloudflared URL 자동 갱신** — 재시작 시 발급되는 새 URL 을 frontend `.env` 에 수동 반영하는 흐름 자동화.
- **결과 export** — Preview 시트 내용을 CSV/Excel 파일로 내보내기.
- **작업 보존** — 분석 도중 backend 강제 종료 시 SQLite 진행 결과로부터 사용자 시각의 복귀 흐름 정립.
- **추론 가속** — bf16 17.5 GiB / ~8s/장에서 INT8/INT4 quantization 또는 대체 모델 후보 검토.
- **모델 정확도 — 세대원 / 분리세대 표 혼동** — `direct_payment` 에서 "주민등록표상 세대원" 과 "주민등록표상 세대를 분리한 가족 구성원" 표를 모델이 혼동해 한 표 값을 다른 카테고리에 출력. 개선 옵션(비용/효과 순):
    1. **fewshot 추가** — `direct_payment.yml` 의 `fewshot: []` 에 카테고리 구분이 명확히 드러나는 예시 2~3쌍 작성. 비용 낮음 / 효과 높음 (VLM 카테고리 학습이 가장 직접적).
    2. **프롬프트 강화** — `_qwen_build_prompt` 에 "두 표는 별개이며 혼동 금지" 명시 + 표 제목 그대로 인용. 비용 낮음 / 효과 중간.
    3. **키 네이밍 정리** — `relationship(rrc)` 등 괄호 키를 `relationship_rrc` / `relationship_non_rrc` 로 교체. JSON 출력 안정성 향상. 비용 낮음 / 효과 중간.
    4. **표별 분리 추출** — 표 1개당 별도 prompt 로 predict 후 어댑터가 병합. 호출 N배라 추론 시간 N배. 비용 높음 / 효과 높음 (한 호출에 한 카테고리만 보면 혼동 거의 없음).
    5. **사전 크롭** — 표 영역 좌표를 yml 에 정의하고 backend 가 crop 한 부분만 모델에 입력. 양식별 좌표 측정 필요. 비용 매우 높음 / 효과 매우 높음.

    추천 순서 : 1+2+3 묶음 먼저 (1세션 내 적용/실측 가능) → 부족하면 4 → 양식 수 늘면 5. fewshot 의 user/assistant 예시는 실제 신청서를 보고 사람이 채우는 게 정확하므로 빈 골격만 깔고 사용자 입력 받는 방식이 안전.
